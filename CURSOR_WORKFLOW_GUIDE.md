# Cursor 工作流指南 - 行为准则

## 🎯 核心原则
你是本项目的代码工程师，必须遵守以下行为准则：

### 1. 关键函数保护
- **不要删除或重命名关键函数**：`to_monthly`、`transform_series`、`get_series_data`
- 这些函数是系统的核心，任何修改都要通过测试验证

### 2. 测试驱动开发
- **任何修改都要通过 pytest**
- 新增函数必须带 docstring 和类型注解
- 先写测试，再改实现

### 3. 数据处理标准
- **统一月末频率**：优先 'ME'，不支持时回退 'M'
- **数值列强制 float64**
- **'.'、'NA' 等统一清洗为 NaN**

### 4. 变更范围控制
- **变更范围最小化**：仅编辑指明的文件/函数
- **保持 API 向后兼容**
- 不要影响其他模块

### 5. 自检机制
- **在提交前自检**：对照"验收清单"逐项核对并在代码注释里勾选
- 每次修改先打印"变更计划"，再给 patch
- 最后附"自检对照清单"打勾

## 🔄 标准工作流程

### 步骤1：变更计划
```
变更计划：
- 目标：[具体要解决的问题]
- 范围：[涉及的文件/函数]
- 风险：[可能的影响]
- 测试：[需要验证的功能]
```

### 步骤2：实现修改
- 按照变更计划进行最小化修改
- 保持代码风格一致
- 添加必要的注释和文档

### 步骤3：自检对照
```
自检对照清单：
- [ ] 通过 pytest -q
- [ ] ruff/black/mypy 无关键告警

- [ ] 分组聚合用加权中位数，已应用红线规则
- [ ] 图片在报告中可移动端直显
- [ ] API 向后兼容性保持
- [ ] 变更范围最小化
```

## 📊 打分逻辑的"带护栏升级"

### 组内聚合
- **改用加权中位数**（没有单项权重就当1）
- 避免简单平均被极端值稀释

### 红线规则
- **任一核心指标≥80分 ⇒ 该组分≥60**
- **若核心≥2个≥80 ⇒ 总分≥40**
- 防止高风险被低风险稀释

### 高风险占比惩罚
- **>30%指标>70分 ⇒ 总分+5**
- **>50% ⇒ +10**（封顶100）
- 反映系统性风险

### 新鲜度门槛
- **月度>90天/周度>30天/日度>15天 ⇒ 该指标权重=0**
- 但仍展示并标⚠️
- 避免过期数据误导

### 去重/降相关
- **同组|ρ|>0.7仅保留更前瞻/更稳定的一条**
- 其余权重=0
- 避免重复计算

## 📋 基准选择（内置映射）

### 利差/曲线/压力
- 对比 `crisis_median` 或 `p25`
- 越高越危险

### 波动/恐慌
- 对比 `p75`
- 越高越危险

### 水平类利率
- 用2000年后样本分位或实质利率
- 且权重减半

### 同比向下为险
- 对比 `p25`（方向取反）
- 如产出/就业

## 🌐 报告生成要求

### 产出两份
1. **report.md**（内部读）
2. **report.html**（自包含，把figures/*.png转成base64）

### 移动端直显
- HTML内嵌base64：`<img src="data:image/png;base64,...">`
- 或https绝对链接：`<img src="https://...">`

### 自动校验
- 报告生成后自动校验图片`<img>`可读
- 检查src是否为`data:image/`或`https://`
